# 2026-04-27 账号池工具栏筛选改造计划 v01

## 背景

当前账号池工具栏只支持 `全部来源 / AUTH FILE / API KEY` 三个并列按钮：

1. 来源筛选占用了较多横向空间
2. “来源”与“可用性/异常”是不同维度，但还没有独立表达
3. 用户关闭页面后，筛选配置不会保留

本轮需要把“全部来源”改造成统一菜单入口，并补齐额度与异常两个可持久化筛选条件。

## 目标

1. 将来源筛选改为单一菜单入口，按钮文案改为“显示”
2. 菜单内支持来源单选：`全部 / AUTH FILE / API KEY`
3. 菜单内新增状态筛选：
   - `仅显示有最长窗口额度`
   - `仅显示异常账号`
4. 将上述筛选配置写入本地存储，页面下次打开时自动恢复

## 非目标

1. 不新增独立高级筛选弹窗
2. 不持久化搜索词
3. 不持久化批量选择状态
4. 不改变账号卡片的额度展示结构

## 关键边界

### 来源筛选

- `source` 维持单选：`all | auth-file | api-key`

### 最长窗口额度筛选

- 仅对 `auth-file + codex` 账号生效
- `api-key` 账号在开启此筛选时直接排除
- 当 quota 仅有一个窗口时，该窗口即视为最长窗口
- 当 quota 有多个窗口时，优先取 `weekly / *-weekly`
- 若无 `weekly`，回退到当前展示窗口中的最后一个窗口
- 只有该窗口存在且 `remainingPercent > 0` 时，才算“有最长窗口额度”
- `loading / error / empty / 无窗口` 统一视为“不满足该筛选”

### 异常账号筛选

- 保留“仅显示异常账号”语义
- 默认将以下账号视为异常：
  - `disabled === true`
  - `rawAuthFile.unavailable === true`
  - `status` 不属于 `ACTIVE / CONFIGURED / LOCAL`

## BDD 场景

### 场景 1：来源筛选改为菜单

- Given 用户进入账号池页面
- When 用户打开“显示”菜单
- Then 菜单内应展示来源单选项：`全部 / AUTH FILE / API KEY`
- And 点击后应立即更新列表

### 场景 2：最长窗口额度筛选

- Given 页面存在多个 `auth-file` 账号，且额度窗口存在单窗口和多窗口两种情况
- When 用户开启“仅显示有最长窗口额度”
- Then 仅保留最长窗口 `remainingPercent > 0` 的 `auth-file + codex` 账号
- And `api-key` 账号不出现在结果中

### 场景 3：异常账号筛选

- Given 页面存在正常账号、disabled 账号、状态异常账号
- When 用户开启“仅显示异常账号”
- Then 仅展示异常或不可用账号

### 场景 4：筛选组合

- Given 用户选择 `AUTH FILE` 且开启“仅显示异常账号”
- When 列表刷新
- Then 仅展示来源为 `auth-file` 且满足异常条件的账号

### 场景 5：本地持久化

- Given 用户已修改来源与状态筛选
- When 用户关闭页面后再次进入账号池
- Then 页面应恢复上次保存的筛选配置

## 实现步骤

1. 扩展账号筛选状态模型，拆分来源与状态筛选
2. 先补 `accountSelectors` 与筛选持久化测试，并确认失败
3. 实现筛选判定与本地存储读写
4. 将 `AccountsToolbar` 改为菜单式交互，并接入新状态
5. 运行相关单测与类型检查
6. 更新记忆并执行 `qmd update`、`qmd embed`

## 验收

1. `AccountsToolbar` 不再展示三个并列来源按钮
2. 菜单内可配置来源与两个状态筛选
3. 最长窗口额度筛选兼容单窗口与多窗口 quota
4. 异常账号筛选能命中 disabled / unavailable / 非正常状态账号
5. 刷新或重开页面后，筛选配置保持不变
