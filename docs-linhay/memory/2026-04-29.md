## Nolon 会话管理研究空间与 debate

- 新建了研究空间 `docs-linhay/spaces/20260429-nolon-session-management/`，用于承载本轮对参考项目 `nolon` 会话管理实现的研究。
- 已产出 debate 纪要：`docs-linhay/spaces/20260429-nolon-session-management/debate/20260429/nolon-session-management/20260429-nolon-session-management-v01.md`。
- 当前阶段性结论：
  - `nolon` 的会话对外口径主要是 `CodexSessionStore` 聚合出的 `CodexSessionRecord` / snapshot，而不是单一 JSONL 或单一 SQLite。
  - 列表首屏采用 `projection cache -> skeleton -> snapshot stream -> reconcile` 的分层加载链路。
  - `rewrite` 是 preview、rollout 文件改写、state db 改写、一致性校验组成的多介质链路，不是事务性原子操作。
  - CLI 与 UI 共用 rewrite 引擎，但 grouping 口径已出现漂移：CLI 仍保留 `time-project`，当前 SwiftUI ViewModel 只剩 `project/provider`。
- 继续深挖后补充确认：
  - `summary` 在当前 `CodexSessionStore` 首屏链路里被明确置空，测试也约束“不从 rollout 正文提炼摘要”。
  - 搜索仅覆盖 `title / displayID / provider / summary / cwd` 这些展示字段，不直接依赖 usage index 或全文索引。
  - `usage index` 的真实职责是 usage 聚合、usage 排序预热与 timeline 复用；列表主快照仍由 projection cache 承担。
  - `time-project` 并非纯文档残留：2026-04-11 设计曾要求 UI/CLI 对齐该模式，但 2026-04-15 的 `project-first` 重构已把 SwiftUI UI 收缩为 `project/provider`，CLI 没同步收口。
  - `rewriteProviders` 的实际执行顺序是 `preview -> live rollout -> archived rollout -> state db -> invalidate cache -> verify`；rollout 侧只改首个 `session_meta` 行中的 `model_provider`，state db 侧只改 `threads.model_provider`。
  - `rewrite` 不是事务性原子操作；测试覆盖了“rollout 已改成功但 SQLite 更新失败”的部分成功场景，系统通过 `failures + verify` 暴露不一致，而不是自动回滚已写入的 rollout。
  - 首屏时序已进一步确认：`cached snapshot` 命中且 `clean + fresh` 时可直接短路 reconcile；否则先展示缓存，再走稳定 snapshot；只有缺缓存时才退回 `skeleton -> stream -> fallback snapshot`。
  - 前台 autoresearch 已将 README 收口到“完整分析完成”：
    - `snapshotStream` 的 delta 语义与 `removedSessionIDs` 的最终批次删除逻辑已经补齐。
    - `logicalUsageKey` 与 forked rollout post-fork usage dedupe 的双层去重语义已经补齐。
    - 已新增一张分层边界矩阵，明确 rollout / state sqlite / projection cache / usage index / ViewModel / CLI 各自职责。
    - 随后又补了两层收口：一张“结论 -> 代码/测试证据”矩阵，以及一条 2026-04-11 到 2026-04-21 的实现演进时间线，方便后续回看时快速判断哪些是当前语义、哪些是历史沉积。
    - 继续下钻后，又补了一段“迁移/复用指南”：把 `nolon` 的 session 管理模式和 GetTokens 当前 `usage_local.go` 的本地 usage index 实现对齐，明确哪些模式可借，哪些产品边界不能直接照搬。

## Usage Desk 本地投影总量差异归因

- 已确认 `Usage Desk -> 本地投影用量 -> 全部` 与 `nolon` 数值不一致，根因不是前端汇总公式，也不是 GetTokens 的 SQLite 索引丢数。
- 实测 GetTokens 旧实现只扫描 `~/.codex/sessions/**/*.jsonl`，因此本机原始全扫与索引库都稳定落在约 `185.67 亿`。
- `nolon` 本机索引库额外纳入了 `~/.codex/archived_sessions/**/*.jsonl`；其 `usage_entries` 明显更多，导致总量跃升到约 `364 亿` 量级。
- 已在 `APP 层` 修复扫描边界：`GetCodexLocalUsage` 现在同时扫描 `sessions/` 与 `archived_sessions/`，并新增回归测试覆盖 archived rollout 被纳入统计的场景。
- 本轮回归已通过 `go test ./internal/wailsapp/...` 与 `frontend npm run typecheck`。

## Usage Desk APP 层缓存与后台增量刷新

- 已确认旧行为存在产品偏差：`H5` 每次刷新页面都会重新调用 `GetCodexLocalUsage` 并触发 `APP 层` 文件清单校验，这虽然复用了 SQLite 索引，但用户仍会看到“重新扫描样本”。
- 当前裁定已经改成 `APP 层` 持有 `LocalProjectedUsage` 进程内快照缓存：
  - `H5` 正常页面刷新默认直接读取 `APP` 当前缓存
  - 只有点击 `刷新索引 / 重建索引` 才显式触发新的增量刷新 / 重建
- `APP 层` 新增了后台分钟级增量刷新编排：
  - 定时器由 `APP` 自己驱动，不下放到 `H5`
  - 定时刷新继续复用本地 SQLite 增量语义，不改成前端轮询
  - 刷新完成后通过 `usage-local:updated` 事件把最新快照推回 `H5`
- `Settings` 已新增本地投影后台刷新间隔设置，当前允许：
  - `5 分钟`
  - `15 分钟`
  - `30 分钟`
  - `1 小时`
