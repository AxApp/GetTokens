# 2026-05-02 Request Orchestration Menu

## 背景
本轮方向已从“账号池页面改造”调整为“新增一个独立菜单承载请求编排”。需求核心不再是账号池信息架构，而是给请求路由、provider 激活、编排策略和相关控制入口一个独立工作区，避免继续塞进账号池语义里。

当前仓库里与请求分发相关的能力分散在：

- 账号轮动与 provider/workspace 入口
- 状态页中的 relay routing 配置
- sidecar 现有 `round-robin / fill-first / session-affinity`
- 独立代理池页面与既有账号级 `proxy-url`

这轮进一步明确页面心智：它不是“表单堆叠的配置页”，而是一个**泳道模式的请求编排工作台**，用可视化链路表达请求从入口进入、如何被分配到账号组、每个账号是否走代理、最终从哪个出口发出。

## 目标
为“请求编排”建立一轮独立空间，承接新菜单的需求边界、泳道交互结构、实现计划、截图和验收记录。

本轮首先收敛以下问题：

- 请求编排是否作为独立一级菜单出现
- 页面是否以泳道方式表达完整请求链路，而不是拆成零散卡片
- 哪些能力属于 APP 层编排，哪些只是 sidecar 已有状态透出
- `请求入口 -> 账号组 -> 账号代理 -> 出口` 四段分别展示什么、能编辑什么
- 每个账号需要声明支持哪些 CLI，以及如何把入口模型映射到该账号真实可用模型
- provider lock / 激活唯一 provider 应作为哪一层泳道动作出现
- 实现完成后需要哪些自动化验证与桌面验收

## 范围
- 新菜单“请求编排”的信息架构、入口定义与页面边界
- 与请求编排直接相关的前端页面、feature、状态组织和交互闭环
- 与请求编排页面直接相关的 Wails 接口编排
- 本轮请求编排需求的计划文档、截图和 debate 记录

若本轮实现涉及以下内容，也纳入范围：

- relay routing 配置编辑入口重组
- provider 激活 / 唯一 provider 编排入口
- 账号组与账号参与关系的可视化编排
- 账号级代理模式展示与编辑入口
- 账号级 `支持的 CLI` 配置
- 账号级 `模型映射 / alias` 配置与可视化摘要
- 出口视角的链路预览与结果摘要
- 请求分发相关的页面文案、空状态和工作区拆分

- 不把该需求继续挂在账号池菜单下实现
- 不在本轮默认修改 CLIProxyAPI 原生策略枚举
- 不把它做成单纯“基础路由配置表单”页面
- 不把代理池节点 CRUD 直接搬进本页；代理池仍是独立页面

## 当前需求

### 页面定位

`请求编排` 是一个新的一级菜单，职责是把“请求如何被组织和发出”收口到一个独立工作台中。

它不是：

- 账号资产管理页
- provider 新增/编辑页
- 代理池管理页
- sidecar 原生策略枚举展示页

它是：

- 请求链路编排页
- 当前生效路径预览页
- APP 层编排动作的执行入口

### 页面主模型：泳道模式

页面采用固定从左到右的四段泳道：

1. `请求入口`
2. `账号组`
3. `账号代理`
4. `出口`

每条泳道都展示“当前状态 + 可操作项 + 对下一段的影响”。

### 泳道 1：请求入口

这一段负责表达“什么请求会进入这条编排链路”。

第一阶段建议至少包含：

- 当前入口类型
  - `本地 Codex`
  - `Relay/OpenAI compatible`
- 当前 CLI 上下文
  - `codex`
  - `claude`
  - `gemini-cli`
  - `openai-compatible`
- 当前默认模型 / provider 上下文
- 是否存在显式 provider lock
- 当前基础路由参数摘要
  - `round-robin / fill-first`
  - `session-affinity`
  - `request-retry`

这一段的动作边界：

- 允许查看当前入口上下文
- 允许跳转到更详细的基础路由配置
- 允许看到当前入口模型会如何参与后续账号兼容性判断
- 不在这一段直接管理账号资产

### 泳道 2：账号组

这一段负责表达“请求进入后，会落到哪一组账号/provider 上”。

第一阶段建议固定为“组视角”，而不是直接平铺所有账号：

- `Codex 组`
- `OpenAI-Compatible Provider 组`
- 后续如有需要再扩更多组

每个组卡至少展示：

- 组名
- 当前是否参与
- 组内可用对象数量
- 当前是否被锁定为唯一活跃组
- 排除原因（若被禁用或被其他编排模式排除）

这一段的主动作：

- `激活为唯一提供组`
- `恢复普通参与`
- 查看组内对象详情

说明：

- 如果用户激活唯一 provider，本质上是 APP 层编排动作，不要求 sidecar 新增原生 `strategy`
- 页面需要明确这是“唯一组/唯一 provider 激活”，不是 sidecar 原生第三种轮动策略

### 泳道 3：账号代理

这一段负责表达“组内每个账号是否走代理，以及走哪种代理模式”。

这一段的核心对象是账号级代理语义，而不是代理池节点本身。

代理模式对齐现有规格，固定为：

- `inherit`
- `direct`
- `custom`

语义对齐 `docs-linhay/dev/20260429-gettokens-proxy-spec-v01.md`：

- `inherit`：不写显式 `proxy-url`
- `direct`：显式直连
- `custom`：携带明确 `proxyUrl`

这一段建议展示为“账号行卡片”：

- 账号名称
- 所属组
- 当前代理模式
- 当前代理来源
  - 继承默认
  - 直连
  - 自定义代理
- 若是 `custom`，展示代理摘要
- 支持的 CLI 列表
- 模型映射摘要
  - 入口模型名
  - 实际目标模型 `name`
  - 展示别名 `alias`（若存在）
- 当前是否兼容入口 CLI / 模型

这一段建议支持的动作：

- 切换账号代理模式
- 选择自定义代理
- 编辑账号支持的 CLI 集合
- 编辑账号模型映射
- 快速跳转到代理池页面查看或维护代理节点

这一段不负责：

- 在本页直接做代理池节点新增、删除、导入

模型映射语义先按“入口模型 -> 账号真实模型”理解，尽量复用仓库现有 `models[].name / models[].alias` 表达，不单独发明新的映射字段名。

### 泳道 4：出口

这一段负责表达“最终请求会从什么出口发出”。

第一阶段至少需要展示：

- 当前出口类型
  - 直连
  - 代理出口
  - 继承环境
- 最终出口摘要
  - provider / baseUrl
  - proxy mode
  - proxy url 摘要（如存在）
- 当前参与出口的对象数量

这一段更偏结果预览，而不是设置页：

- 用于确认上一段编排后的最终生效路径
- 用于帮助用户判断配置是否符合预期

### 页面区块补充

除四段泳道外，页面底部或右侧允许增加一个收口区：

- `当前生效预览`
- `最近一次编排变更`
- `恢复默认编排`

但这些是辅助区，不应替代四段泳道本身。

## 领域边界

### 1. APP 层编排优先

本轮新增的“唯一 provider / 唯一账号组”语义，默认先按 APP 层编排实现，不要求 CLIProxyAPI 增加新的原生 `routing.strategy`。

### 2. 账号组与账号资产分层

页面第二段展示“账号组”，第三段展示“账号级代理”，不要把“选择组”和“编辑单个账号代理”混成一层。

### 3. 账号兼容性必须显式可见

当入口 CLI 或入口模型变化时，页面需要能够明确表达：

- 哪些账号支持当前 CLI
- 哪些账号虽然在组内，但没有对应模型映射
- 哪些账号因此被排除在本次有效出口之外

### 4. 代理池继续独立

请求编排页面只消费代理结果和代理模式，不承载代理池本体管理；代理池仍保留独立一级页面。

### 5. 页面必须表达链路，而不是只表达配置

用户进入本页后，第一感知应是“请求怎么走”，不是“我在编辑一堆 YAML/JSON 配置”。

## BDD 场景

### 场景 1：进入请求编排页时看到完整泳道

- Given 用户进入应用主界面
- When 用户点击一级菜单 `请求编排`
- Then 页面按从左到右展示：
  - `请求入口`
  - `账号组`
  - `账号代理`
  - `出口`
- And 用户无需展开调试信息就能理解当前请求链路

### 场景 2：激活唯一 provider 组

- Given 页面已展示多个可参与请求的账号组
- When 用户在 `账号组` 泳道中选择某个 provider 组并点击“激活为唯一提供组”
- Then 该组显示为唯一活跃组
- And 其他组显示为被排除
- And `出口` 泳道的结果预览同步更新

### 场景 3：按账号切换代理模式

- Given 用户已进入 `账号代理` 泳道
- And 当前组内存在多个账号
- When 用户把某个账号的代理模式从 `inherit` 改为 `custom`
- Then 页面要求用户选择一个明确代理
- And 该账号行展示新的代理来源与摘要
- And `出口` 泳道同步反映该变化

### 场景 4：账号显式直连覆盖默认代理

- Given 当前存在全局继承或默认代理背景
- When 用户把某个账号设置为 `direct`
- Then 该账号必须被标记为显式直连
- And 该账号不再继承默认代理
- And 出口预览中明确显示该账号走直连

### 场景 5：没有可用组时给出空状态

- Given 当前没有任何可参与请求的账号组
- When 用户进入 `请求编排`
- Then 页面展示明确空状态
- And 空状态说明缺的是哪一层：
  - 没有账号组
  - 没有可用账号
  - 没有可选出口

### 场景 6：按账号声明支持的 CLI

- Given 用户已进入 `账号代理` 泳道
- And 某个账号当前只支持 `codex`
- When 用户为该账号补充支持 `claude`
- Then 该账号卡片应显示新的 CLI 支持集合
- And 当入口切换到 `claude` 时，该账号不再因 CLI 不兼容被排除

### 场景 7：按账号配置模型映射

- Given 当前入口模型为 `gpt-5.5`
- And 某个账号存在 `models[].name / models[].alias` 配置
- When 用户把该账号的入口模型映射调整为 `gpt-5.5 -> gpt-5.4-mini`
- Then 账号卡片显示最新映射摘要
- And `出口` 泳道展示的最终目标模型同步更新

### 场景 8：入口模型无映射时标记账号不可用

- Given 当前入口 CLI 与账号兼容
- And 该账号没有当前入口模型的映射关系
- When 页面刷新当前编排结果
- Then 该账号应被标记为“缺少模型映射”
- And `出口` 泳道不把该账号计入可用出口对象

### 场景 9：恢复普通编排

- Given 用户此前启用了唯一 provider 组
- When 用户点击“恢复默认编排”
- Then 页面退出唯一组模式
- And 相关组的参与状态恢复到默认编排结果
- And 出口预览同步恢复

## 开放问题

1. 第一阶段的“账号组”是否固定只展示 `Codex` 与 `OpenAI-Compatible` 两组，还是允许更细的 provider 组展开？
2. 第三段“账号代理”里的 `custom` 模式，是直接选现有代理池节点，还是先只允许输入 `proxyUrl`？
3. “请求入口”是否需要区分不同客户端来源，例如本地 Codex 与 Relay API 调用？
4. `支持的 CLI` 是多选集合，还是区分“主要 CLI + 兼容 CLI”两层语义？
5. 模型映射是全局按账号维护，还是需要细化到“按 CLI 维护不同映射表”？
6. 入口模型映射是否严格复用现有 `models[].name / models[].alias`，还是需要额外的“入口模型别名”层？
7. “出口”是否需要展示真实探测能力，还是第一阶段只做配置结果预览？
