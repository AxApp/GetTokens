# GetTokens Subagent Delivery Loop

## 背景

这轮 `text-scale-settings` 会话里，需求不是停在“代码改完”就结束，而是实际走到了完整闭环：

1. 主控 agent 先整理 `space` 边界与验收标准。
2. 把实现拆给多个 `subagent` 并行推进。
3. 主控 agent 负责持续集成、回归、截图、Wails 绑定层验证、文档写回与最终完成判断。
4. 当用户明确要求“不要在需求结束前停止”后，主控 agent 没有把“截图 / 实机验证”当作可选尾项，而是继续推进到闭环完成。

这说明仓库里已经形成一条稳定可复用的交付模式：**主控 agent 负责监督与收口，subagent 负责具体实现，直到需求完整完成才停止。**

## 适用场景

以下任一情况，建议直接采用这套流程：

1. 用户明确要求“由 subagent 去做，我负责监督到完成”。
2. 一个需求会持续多轮，且同时包含实现、验收、截图、文档等多个阶段。
3. 需求可拆成多个相对独立的子任务，适合并行推进。
4. 主控 agent 需要把更多注意力放在边界、集成和验收，而不是自己写完所有代码。

## 角色划分

### 主控 agent

主控 agent 不是“旁观的项目经理”，而是需求交付的最终责任人。它必须负责：

1. 需求边界与验收标准
2. `space` / `branch` / `worktree` 对齐
3. subagent 拆分与调度
4. 持续集成与冲突处理
5. 自动化验证
6. Wails / 桌面验收
7. 截图与其他验收产物归档
8. docs / memory / `qmd` 写回
9. 最终完成判断

### subagent

subagent 负责 bounded implementation work，不负责最终完结判断。它适合承担：

1. 单个代码切片的实现
2. 指定文件集合内的修复或重构
3. 只读探索与现状收集
4. 测试补齐或局部验证

subagent 的输出必须被主控 agent 审阅、集成并纳入统一验收链。

## 标准执行顺序

### 1. 先把需求收成可执行边界

主控 agent 先确认：

1. 对应 `space` 是否存在
2. `README.md` 是否已经写清背景、目标、范围、验收标准
3. 是否需要对应 `branch` / `worktree`

如果这些边界不清，先补齐再启动实现。

### 2. 拆 subagent，但只拆可分离责任

主控 agent 应按“写入面互不重叠”的原则拆任务，例如：

1. 基础设施层
2. 设置页与共享组件层
3. 页面级尾项清理
4. 只读探索或归档规范收集

不应把“当前立即阻塞主控下一步”的关键判断完全外包。

### 3. 主控持续集成，不等到最后才合并

主控 agent 在 subagent 返回后应立即：

1. 审阅变更
2. 集成到主线
3. 发现冲突时本地修正
4. 立刻补验证

避免等全部 subagent 返回后再做一次大合并。

### 4. 不把“首批实现”误判为完成

以下状态都不能算需求完成：

1. 代码已经改完，但还没跑测试
2. 测试通过，但还没做 Wails / 桌面验证
3. 桌面验证做了，但还没归档截图
4. 截图有了，但文档和 memory 没写回
5. 文档写了，但 `qmd update` / `qmd embed` 没执行

真正完成必须跑完整条链。

### 5. 标准收尾链

主控 agent 收尾时应默认检查以下项目：

1. 代码已经集成到主线
2. 自动化验证已执行
3. Wails / 桌面实机验证已执行
4. 截图或其他验收产物已归档
5. `space README` 已更新当前状态
6. `docs-linhay/memory/YYYY-MM-DD.md` 已写回关键结论
7. `qmd update`
8. `qmd embed`
9. 必要时 `docs-linhay/scripts/check-docs.sh`

只有这些都闭合后，才向用户报告需求完成。

## 停止条件

主控 agent 只有在以下三种情况下可以停止：

1. 需求完整闭环，所有验收链都已完成
2. 用户明确要求暂停或切换
3. 存在当前环境内无法自行解决的具体 blocker，并已明确说明原因、影响和剩余工作

“代码已经提交”不是停止条件。  
“只剩截图了”也不是停止条件。  
“只差文档写回”同样不是停止条件。

## Wails 场景下的额外约束

对 GetTokens 这类 Wails 桌面项目，还需要额外注意：

1. 裸 `vite` 页面不能替代 Wails runtime 验证。
2. 如果功能依赖 Wails bindings，必须优先验证 `Wails bridge` 或真实桌面窗口。
3. 如果系统权限限制导致无法自动抓取真实桌面窗口，应如实记录，并用当前环境中最接近真实运行态的证据链补足，例如：
   - Wails dev bridge 页面
   - 原生进程运行证据
   - 绑定层截图

## 这次会话沉淀出的固定结论

1. “主控 agent 监督 + subagent 实做”是可行的，但前提是主控 agent 不能把自己降级成只转发消息。
2. 完整需求的停止点应放在“验收链闭合”而不是“实现链闭合”。
3. `screenshots / 桌面验证 / docs / memory` 不应被视为附加工作，而应视为需求的一部分。
4. 对这种模式，最合适的沉淀层级是：
   - 具体执行顺序：项目级 `skill`
   - repo-wide 强约束：`AGENTS.md`
   - 解释性说明：`docs-linhay/dev/`

## 推荐口令

如果后续用户希望继续按这套方式走，可以直接使用下面这类表达：

1. “这个需求按 subagent 方式完整做完，你负责监督到闭环。”
2. “不要停在首批实现，继续推进到验收、截图、文档都完成。”
3. “主控 agent 负责调度和收口，subagent 负责实现。”

这些表达在 GetTokens 仓库里，应直接触发本流程。
