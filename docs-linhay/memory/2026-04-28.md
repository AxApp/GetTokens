## GetTokens Usage Dual Source Debate
- 已新建需求空间 `docs-linhay/spaces/20260428-gettokens-usage-dual-source/`，专门讨论 GetTokens 的双轨用量统计设计。
- debate 已达成共识：GetTokens 后续应拆成三条并列信息，而不是一个“用量大桶”：
  - `ObservedRequestUsage / 真实请求用量统计`
  - `LocalProjectedUsage / 本地用量统计`
  - `QuotaSnapshot / 配额快照`
- 当前 `GetUsageStatistics` 继续只代表 sidecar `/usage` 观测链路；它应被重新命名和展示为“请求观测/稳定性”，而不是 provider 全量消耗。
- `quota` 继续独立域；即便底层有时吃 cached usage snapshot，最终语义仍只是额度窗口，不属于 usage fact。
- 已按本机 `~/.codex` 样本复核：`session_meta / turn_context` 里只有 `cwd / git / model_provider / source / model` 等字段，没有可稳定映射到 GetTokens `auth-file` 的 `authIndex / account_id / email / token` 类标识，因此 `Codex local usage` 当前无法归因到 `auth-file`；首版只能做 provider/global 级视图。
- 已进一步收紧本地 usage 边界：`LocalProjectedUsage` 必须先按 `provider` 分域统计，不能把 `Codex / Claude / Gemini` 这类本地事件混成一个总桶；即使首版只落地 `Codex`，数据模型、Wails API 和前端状态也必须保留 `provider` 维度。
- 前端 IA 已进一步裁定：v1 入口仍放在 `Accounts`，但要新增独立的 `Usage Desk` 子工作区或等价页面级分析区；账号卡片继续只展示资产状态、`recent health` 与 quota，不承载本地 usage 明细或趋势图。
- 已在当前 space 根目录落一份 `HTML` 设计稿：`usage-dashboard-design-v01.html`，用于承载 `provider` 过滤、`sourceKind` 分段切换和双轨用量的独立分析面。
- 已归档 debate：`docs-linhay/spaces/20260428-gettokens-usage-dual-source/debate/20260428/gettokens-usage-dual-source-debate/20260428-gettokens-usage-dual-source-debate-v01.md`。
