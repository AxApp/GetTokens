# “整理”会话触发动作与跨项目移植说明

## 1. 背景

在 GetTokens 仓库里，用户说“整理”不是普通的聊天总结口令，而是一次正式的会话沉淀动作。

它的目标不是复述这轮做了什么，而是把这轮工作里已经证明可复用、后续还会重复出现的模式，归档到正确层级，避免知识继续只停留在会话里。

当前规则来源：

1. `AGENTS.md`
2. `.agents/skills/gettokens-session-skill-distill/SKILL.md`
3. `.agents/skills/gettokens-ops-governance/SKILL.md`

## 2. 触发条件

满足以下任一条件，就应触发“整理”流程：

1. 用户明确说“整理”。
2. 当前语境明显指向刚完成的一轮工作会话，需要做收尾沉淀。
3. 同一轮会话里反复出现同类排障、交付、验收或文档动作。
4. 需要判断某个模式应该沉淀为 `skill`、`docs`、`memory`，还是升级到 `AGENTS.md`。

补充约束：

1. 如果用户说“整理”且语境明确，不需要额外追问是否要沉淀。
2. 默认直接进入模式提炼与写回流程，而不是只输出一段总结文字。

## 3. 实际会触发的动作

当“整理”被触发时，标准动作顺序如下：

1. 先回看本轮工作，抽取真正可复用的模式、边界、排障路径或交付动作。
2. 把临时内容排除掉，例如一次性猜测、单次提交号、临时样本数据、仅本轮有效的口头表述。
3. 判断沉淀层级：
   - 只在本轮出现且不会复用的，丢弃
   - 后续还会重复的，优先沉淀到项目级 `skills`
   - 跨需求、跨会话都稳定成立的，写入 `docs-linhay/dev/`
   - 关键决策、行动项、里程碑、风险结论，写入 `docs-linhay/memory/YYYY-MM-DD.md`
   - 只有 repo-wide 且长期稳定的硬规则，才升级到 `AGENTS.md`
4. 若发现缺少执行入口，优先新增或更新 `.agents/skills/`，而不是先扩写 `AGENTS.md`。
5. 完成文档与记忆写回后，执行 `qmd update` 与 `qmd embed`，保证可检索。

## 4. 不会触发的动作

“整理”不等于以下行为：

1. 不等于自动生成冗长会议纪要。
2. 不等于把整轮聊天原样复制进文档。
3. 不等于默认修改 `AGENTS.md`。
4. 不等于把所有细节都升级成 `skill`。
5. 不等于只做口头总结而不写回仓库。

## 5. 产出物清单

一次完整的“整理”通常会产出以下一种或多种结果：

1. 项目级 `skill`
   - 适用于可重复执行的项目动作、稳定排障路径、固定交付套路
2. `docs-linhay/dev/*.md`
   - 适用于跨需求复用的技术边界、工作流说明、治理规则解释
3. `docs-linhay/memory/YYYY-MM-DD.md`
   - 适用于关键决策、行动项、里程碑、风险结论
4. `AGENTS.md`
   - 仅适用于 repo-wide、长期稳定、必须强制执行的规则
5. `qmd` 索引更新
   - 通过 `qmd update` 与 `qmd embed` 让新知识可检索

## 6. 层级判断矩阵

可直接复用下面这套判断方式：

| 问题 | 结论 |
|------|------|
| 这条信息只对当前会话有效吗？ | 是：不沉淀 |
| 这条信息是稳定执行动作吗？ | 是：优先沉淀到 `skill` |
| 这条信息是跨需求技术事实或流程解释吗？ | 是：写 `docs/dev` |
| 这条信息是关键决策或风险结论吗？ | 是：写 `memory` |
| 这条信息是否 repo-wide 且长期稳定？ | 是：再考虑写 `AGENTS.md` |

核心原则只有一条：先沉淀到最小、最贴近实际复用边界的层级，避免过早升级到 `AGENTS.md`。

## 7. 建议执行模板

如果要把这套流程移植到其他项目，建议保留以下固定步骤：

1. 定义一个明确口令，例如“整理”。
2. 在项目规则里写清楚这个口令的含义：
   - 它代表“会话沉淀”
   - 它不是普通总结
3. 约定沉淀层级：
   - `skills`
   - `dev docs`
   - `memory`
   - `AGENTS`
4. 约定升级顺序：
   - 先 `skill`
   - 再 `docs/memory`
   - 最后才是 `AGENTS`
5. 约定索引同步动作，例如：
   - `qmd update`
   - `qmd embed`
6. 约定“不纳入”的内容，避免文档膨胀。

## 8. 跨项目最小接入要求

如果其他项目也想用这套机制，至少需要具备下面四个基础件：

1. 一个项目规则入口
   - 例如 `AGENTS.md`
2. 一个项目级可复用动作目录
   - 例如 `.agents/skills/`
3. 一个长期文档目录
   - 例如 `docs/dev/`
4. 一个记忆或决策日志目录
   - 例如 `docs/memory/`

如果缺少其中任意一项，“整理”就容易退化成只输出聊天总结，而不是形成可持续维护的项目资产。

## 9. 可直接移植的规则片段

其他项目可以直接参考下面这段规则表达：

> 当用户明确说“整理”且语境指向刚完成的一轮工作会话时，默认触发一次会话沉淀流程：先提炼可复用模式，再按稳定性边界决定是否同步更新项目级 skills、研发文档、memory；只有当规则已经上升为 repo-wide、长期稳定的约束时，才更新 AGENTS。完成后执行索引同步命令，保证新知识可检索。

实际落地时，只需要把其中的目录名替换成目标项目自己的文档结构即可。

## 10. GetTokens 当前结论

在 GetTokens 当前治理下，用户说“整理”会默认触发以下闭环：

1. 提炼会话中的可复用模式
2. 先补或更新项目级 `skills`
3. 同步写入 `docs-linhay/dev/` 与 `docs-linhay/memory/`
4. 仅在 repo-wide 长期稳定时更新 `AGENTS.md`
5. 执行 `qmd update` 与 `qmd embed`

因此，“整理”在本仓库里的实际含义可以概括为：

`skill / docs / memory / AGENTS / qmd` 五件套的会话沉淀动作。
